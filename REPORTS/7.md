# EvalFlow Sprint 6 Report: SSLCommerz Production Fix, Grade Page Redesign & Access Control

---

## Overview
Sprint 6 is a hardening sprint focused on resolving the SSLCommerz payment gateway integration from end to end, redesigning the evaluator's grading page for usability, and introducing proper credit-based access control to the student exam flow. By the end of this sprint, the full payment lifecycle works against the real SSLCommerz sandbox, the evaluator can always see and click "Save All Marks", and students are blocked from viewing question papers without sufficient wallet balance.

## Features Implemented

### 🔧 SSLCommerz Integration — End-to-End Fix (5 Commits)
The SSLCommerz payment flow was non-functional after Sprint 5 because clicking "Pay with SSLCommerz" silently redirected to the homepage instead of the SSLCommerz sandbox page. This sprint identified and fixed **three root causes**:

1. **Wrong Default Credentials** — The hardcoded fallback password was `testpassword`. The actual SSLCommerz sandbox test password is `qwerty09876@SSL`. Credentials were moved to `config/services.php` backed by `.env` variables (`SSLCOMMERZ_STORE_ID`, `SSLCOMMERZ_STORE_PASSWORD`).
2. **Empty `GatewayPageURL` Redirect** — When SSLCommerz rejected the credentials, it returned `{"status":"FAILED","GatewayPageURL":""}`. The old code used `isset()` which returns `true` for empty strings, so `redirect('')` silently went to `/`. Fixed by switching to `!empty()`.
3. **`APP_URL` Mismatch** — `APP_URL` was `http://localhost`, but the dev server runs on `http://127.0.0.1:8000`. SSLCommerz callback URLs were unreachable. Fixed in `.env`.
4. **Callback Routes Behind Auth Middleware** — SSLCommerz POSTs back from a cross-origin domain. Modern browsers drop session cookies on cross-origin POST due to `SameSite=Lax`. The success/fail/cancel callback routes were moved outside the `auth` middleware group.
5. **IPN Validation** — The `success` callback now calls the SSLCommerz Validation API (`/validator/api/validationserverAPI.php`) server-side using the `val_id` to cryptographically verify the transaction was not spoofed before crediting the wallet.

### 🎨 Evaluator Grade Page — Full Redesign
The grading page had a critical usability bug: the "Save All Marks" submit button was pushed off-screen by the question list and the "Previously Saved Marks" summary, making it impossible to click.

**Redesign highlights:**
- **Sticky Footer** — The submit button and running total are now pinned to the bottom of the right panel, always visible regardless of how many questions exist.
- **Light-Themed Grading Panel** — The right panel uses a clean white/gray background for better readability while grading. The left PDF viewer remains dark-themed.
- **Independent Scrolling** — The questions list scrolls independently. The header (student info, max marks) and footer (total + submit) remain fixed.
- **"Previously Saved Marks"** summary moved inside the scroll area so it no longer pushes the submit button off-screen.
- **Full-Height Layout** — The entire page fills the viewport with `height: calc(100vh - 64px)`.

### 💰 Evaluator Withdrawal — Max Amount Validation
- **Server-side max validation** — The withdrawal amount now uses Laravel's `max:{balance}` validation rule. If an evaluator enters more than their available balance, the form is rejected with the message: *"You cannot withdraw more than your available balance (৳X.XX)."*
- **HTML `max` attribute** — The input field also has `max="{{ $user->balance }}"` for browser-native validation.
- **Flash messages** — Success and error flash messages are now properly displayed at the top of the evaluator finance page.

### 🔒 Student — Credit-Gated Exam Access
- **Dashboard Button Disabled** — If a student's wallet balance is less than the exam's `total_marks`, the "View Paper & Upload" button is visually greyed out (`opacity: 0.6`, `cursor: not-allowed`) with a red "Needs {X} TK" label below it.
- **Server-Side Guard** — Even if a student manually navigates to `/student/exams/{id}`, the `showExam()` controller method checks `auth()->user()->balance < $exam->total_marks` and redirects back with an error message. Students who have already submitted are exempt (they can still view their submission).

## How to Test (End-to-End)

1. **SSLCommerz Payment Flow:** Log in as Student → My Wallet → Enter 500 TK → Click "Pay with SSLCommerz" → Redirects to real SSLCommerz sandbox page → Use test card `4111111111111111`, expiry `12/25`, CVV `111`, OTP `111111` → Success → Redirected back with balance credited.
2. **Credit-Gated Access:** As a student with 0 TK balance, go to Dashboard. The "View Paper & Upload" button should be greyed out with "Needs {X} TK". Trying to visit the URL directly should redirect you back.
3. **Take Exam After Top-Up:** Top up enough credits, then click "View Paper & Upload" (now enabled). Upload your script. Credits are deducted.
4. **Evaluator Withdrawal Limit:** Log in as Evaluator → Earnings → Enter an amount greater than available balance → Submit → See red error: "You cannot withdraw more than your available balance (৳X.XX)."
5. **Evaluator Grade Page:** Open a script for grading → Verify the "Save All Marks" button is always visible at the bottom, even with many questions.
